# DTR Tracking System

A complete Daily Time Record (DTR) monitoring and tracking system with React Native mobile app for students, React.js web admin panel, and Node.js backend API using Firebase Firestore.

## Project Structure

- `backend/` - Node.js API server
- `mobile/` - React Native Expo app for students
- `web/` - React.js admin panel
- `firebase/` - Firebase configuration
- `docs/screenshots/` - UI screenshot assets and references

## Setup Instructions

### 1. Firebase Setup

1. Create a new Firebase project at https://console.firebase.google.com/
2. Enable Firestore Database
3. Create a service account and download the JSON key file
4. Place the JSON file as `backend/firebase-service-account.json`
5. Update `firebase/firebaseConfig.js` with your project config

### 2. Backend Setup

```bash
cd backend
npm install
# Update .env with your JWT secret
npm start
```

### 3. Mobile App Setup

```bash
cd mobile
npm install
expo start
```

#### Mobile API Connectivity (Important)

- For physical devices, avoid using `localhost` as backend URL.
- Set `EXPO_PUBLIC_API_URL` in `mobile/.env` to your current machine IP (example: `http://10.88.111.250:3000`).
- Restart Expo after changing `.env`: `npx expo start -c`
- Ensure backend is running on port `3000` before logging in from mobile.

### 4. Web Admin Setup

```bash
cd web
npm install
npm start
```

## Features

- Student time-in/time-out via mobile app
- Admin management of students and DTR records
- Real-time notifications
- Calendar view with color-coded statuses
- Role-based access control

## Database Rules

Ensure Firestore rules allow read/write for authenticated users as per your security requirements.

## UI Feedback

- Toast messages for quick feedback
- Modals for confirmations
- Animations for successful time-in/time-out
- Empty states:
- "No records yet"
- "No notifications available"

## Backend Automation

- Auto time-out handled in backend (not frontend)
- Ensures functionality even if user is offline

## Performance Optimization

- Notifications: limited to 5 on dashboard
- DTR history: query by month only
- Pagination or lazy loading used for notifications
- Pagination or lazy loading used for records

## Ticket Deletion Rules

- Students can delete tickets only if status is `pending`
- When deleted, related notifications are removed or marked invalid

## User Account Rules

- Admin can create student accounts
- Admin can edit first name and last name
- Admin can reset credentials
- Default credentials username format: `user-<Surname>`
- Default credentials password format: `pass-<Surname>`
- Students can change username and password
- Re-login is required after credential update

## Firestore Security Rules

- Students can only access their own data
- Admin has full access to all records
- Enforced using role field and Firebase Auth UID

## Notifications

- Real-time notifications using Firestore listeners
- Supports read/unread status
- Deduplicates repeated notifications
- Auto time-out notifications are generated by backend

## Mobile Dashboard Features

- User profile icon (top-right) routes to profile page
- Welcome message
- Carousel actions: Time In, Time Out, My Records
- Quick Action card for one-tap guided Time In/Time Out
- Displays last log time for today's attendance activity
- Notification panel shows latest 5 notifications
- "View All" link routes to notifications page
- Navigation flow: Dashboard to Profile, Notifications, and My Records

## Active Session Logic

- A user is considered timed in if `timeIn.length > timeOut.length`
- Backend validates all session states

## DTR Rules

- Multiple time-in/time-out entries are allowed per day
- Cannot time-in twice in a row
- Must time-out before next time-in
- Auto time-out at 12:00 PM (lunch)
- Auto time-out at 8:00 PM (end of day)
- Minimum 1 minute between time-in and time-out
- No time-in/out for future dates
- Students cannot edit past records

## Risks and Mitigations

### 1. Backend Dependency (Critical)

- Auto timeout, validation, and security are enforced in backend logic.
- If backend is down, core DTR workflows are unavailable.
- Mitigation:
- Keep backend health checks and uptime monitoring enabled.
- Add retry handling and clear offline/error states in clients.

### 2. Firestore Cost Risk

- Notifications and calendar/history views can trigger high read volume.
- Mitigation:
- Use optimized queries (month-based filtering, limits, pagination/lazy loading).
- Keep dashboard notifications capped to latest 5 records.

### 3. Data Integrity Risk

- `timeIn` and `timeOut` arrays must remain valid session pairs.
- Mitigation:
- Always validate session state in backend.
- Enforce rules like no double time-in and required time-out before next time-in.

### 4. UX Complexity Risk

- Feature-rich dashboards can become cluttered and confusing.
- Mitigation:
- Keep UI simple, consistent, and action-focused.
- Prioritize clear hierarchy: quick actions, key status, and concise notifications.
